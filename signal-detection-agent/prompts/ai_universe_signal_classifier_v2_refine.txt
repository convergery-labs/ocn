==============================
REFINEMENT RULES
==============================

YOUR ONLY JOB IN THIS PASS IS TO ADJUST THE BASE SCORE USING CONTEXT.
Do not re-score the article from scratch. Stage 1 already decided signal worthiness.
Your pre_verification_score must start from base signal_score and move only when
entity history or web snippets give a concrete reason to change it.

- If entity history is empty AND web snippets are empty: pre_verification_score = base
  signal_score exactly. Set confidence to "low". Do nothing else.
- Upgrade pre_verification_score (move toward signal band) only when context confirms
  a concrete new development, confirms no prior coverage, or web snippet strongly
  corroborates the claim.
- Downgrade pre_verification_score (move toward weak_signal band) only when entity
  history shows the same entity had an identical or very similar event recently.
  Use only signal and weak_signal rows for downgrade decisions — never downgrade
  based on noise rows in entity history.
- Do not downgrade based on web snippets alone - only upgrade or hold with them.
- Do not cross a detection band boundary (signal ↔ weak_signal) unless the context
  evidence is unambiguous. When in doubt, hold the base signal_detection.
- Noise rows in entity history (signal_detection = "noise") are escalation context,
  not downgrade triggers. If prior noise rows described regulatory concerns, financial
  difficulties, competitive pressures, or safety issues for the same entity, and the
  current article reports a concrete event confirming those concerns, treat the noise
  history as corroboration — raise confidence and consider a score upgrade.
  Example: two prior noise rows about "regulators may investigate Company X" followed
  by the current article reporting a formal probe opening → noise history confirms
  escalation, raise confidence to medium or high and support an upgrade if borderline.
- Negative events (fraud, misconduct, short-seller allegations, regulatory enforcement, accounting irregularities, contract terminations, benchmark misrepresentation) are valid signal upgrades when web snippets or entity history corroborate the claim. Do not downgrade a negative event just because it is adverse — the adverse nature of an event is not a reason to lower the score.
- Reconsider category only if web context or entity history clearly suggests a different one.
- When signal_detection is "noise", always set materiality to "none".
- Batch context (other articles in this run): if multiple batch articles share the same
  entity or event as this article, treat this as a corroboration signal - raise confidence
  by one level (low→medium or medium→high) when other evidence is neutral. Do not downgrade
  signal_detection based on batch context alone; it only affects confidence.

==============================
SCORE VERIFICATION
==============================

Before finalising your signal_score, generate 2–3 questions that
directly challenge whether this specific article deserves its
classification. Questions must target the article's actual claims -
not generic checks.

Ask questions that get at:
- Whether the specific claim is confirmed or speculative
  ("reportedly", "is exploring", "plans to", "sources say")
- Whether the named scale, timeline, or counterparty is verifiable
  from the article text or web snippets
- Whether the outcome described has actually happened vs being announced
- Whether a specific entity relationship stated is real or assumed

Bad question (too generic): "Is this AI-related?"
Bad question (wrong scope): "Does this article confirm a direct AI use case?" - category
relevance was already decided in the base pass; do not re-litigate it here.
Good question (article-specific): "Did NVIDIA and Microsoft actually
sign this deal, or is the $10B figure from a single unnamed source?"

Answer each question honestly using the article text, web snippets,
and entity history.

Output pre_verification_score as your refined score based ONLY on the
article + entity history + web snippets - before considering your
verification answers. Do NOT self-apply score_adjustments; the system
computes the final signal_score automatically by summing them.

→ pre_verification_score: number between 0.0 and 1.0

→ verification_qa: 2–3 objects, each with:
   - "question": the specific challenge (~15 words)
   - "answer": honest answer based on article + context (~20 words)
   - "evidence": exact verbatim sentence from the article text, web snippet, or entity history
                 row that drove this answer - copy word-for-word, do not paraphrase.
                 If the evidence comes from a web snippet and a URL was provided in the search
                 results, append it: "Exact quote - source: <url>"
                 Only include a URL if it appeared in the web search results provided to you.
                 Never generate or guess a URL. Use "no direct evidence" if nothing was available.
   - "score_adjustment": number between -0.20 and +0.10

       Stage 1 already read the article and scored it. Its base score already reflects whatever
       the article stated as confirmed fact. Stage 2 only moves the score when context gives a
       reason to change Stage 1's assessment.

       positive (e.g. +0.03 to +0.10): ONLY when a web snippet or entity history resolves
           uncertainty that existed in the article at Stage 1 time - i.e. the article used hedging
           language ("reportedly", "is exploring", "sources say", "plans to") and the web snippet
           confirms the claim is real. Positive = reducing uncertainty, not corroborating certainty.
           Do NOT use positive just because a web snippet agrees with what the article already
           stated as confirmed fact - the base score already reflected that certainty.

       0.00: when the article stated the fact as confirmed AND web agrees, OR when the answer
           is neutral and adds no new information either way.

       negative (e.g. -0.02 to -0.20): when web contradicts the article, when web fails to
           confirm a speculative claim, when the article overstates certainty, or when the answer
           reveals the claim is weaker than Stage 1 assumed.

==============================
NOVELTY
==============================

novelty (string): Where on this story's lifecycle curve does this article land?
Novelty is always decreasing - the first article about an event has maximum novelty;
every subsequent article has less to add.

The unit is the SPECIFIC EVENT THREAD, not the entity. Frequent coverage of a company
generally does NOT lower novelty if this specific claim is new.

Decide by working through this decision tree in order - stop at the first YES:

1. Is the core claim genuinely new - nothing in web search results or entity history
   describes this specific event (same company + same deal/product/announcement)?
      YES → "step_change"

2. Is it a follow-on to a known story, but a key parameter materially changed?
   (timing shifted, scale changed, economics changed, outcome reversed, counterparty changed)
      YES → "meaningful_update"

3. Is it a follow-on with one minor detail added, but the picture is unchanged?
      YES → "incremental_update"

4. Same facts already widely published, nothing new.
      → "repeated_coverage"

novelty_basis: State which branch was taken and what evidence drove it. For each web snippet
that directly supports the novelty decision, include the exact verbatim quote and its source URL
(only if the URL was present in the provided search results - never generate a URL). Cite as many
relevant snippets as apply - do not limit to one. These are external source URLs, not the article's
own URL.
Examples:
- "First appearance of this acquisition; no web results or entity history found."
- "Follow-on to known Mistral story; valuation dropped 30% per web snippet:
  'Mistral's valuation fell to $5B from $6B in latest round' - source: https://..."
- "Third article on same OpenAI partnership; web confirms repeated coverage:
  'OpenAI and Microsoft extend cloud deal' - source: https://...
  'OpenAI Microsoft partnership renewed for five years' - source: https://..."

==============================
CONFIDENCE
==============================

confidence (string): How much to trust this final classification, based on evidence quality.

Assign directly - do not produce a numeric score:

- "high": Clear named entities, article is detailed, base and second-pass agree,
  and you have web snippets or entity history to anchor the claim.
- "medium": Thin article, limited context, borderline base score, or base and
  context mildly disagree.
- "low": No named entities, no web snippets, no entity history, or base was
  deeply borderline with no corroborating context.

confidence_basis: State what specific evidence drove the label. For each web snippet that directly
anchors the confidence decision, include the exact verbatim quote and its source URL (only if the
URL was present in the provided search results - never generate a URL). Cite as many relevant
snippets as apply - do not limit to one. These are external source URLs, not the article's own URL.
Examples:
- "Clear named entities; base and second pass agree. Reuters confirms:
  'NVIDIA and Microsoft signed a $10B Blackwell deal' - source: https://...
  Bloomberg corroborates: 'Deal valued at $10B across Azure data centers' - source: https://..."
- "No named entities; 0 web snippets; base score was borderline at 0.48."
- "Thin article summary only; entity history confirms pattern but article lacks detail."

RULES
==============================

- Assign novelty by working through the decision tree above in order; stop at the first YES. novelty_basis must name the evidence (snippet count, entity history count, specific fact) that determined the branch.
- Assign confidence by reading the checklist above; pick the label that best fits the evidence available. confidence_basis must name the specific evidence (entity count, snippet count, agreement/disagreement with base) that drove the label.

- materiality must be "none" when signal_detection is "noise".
